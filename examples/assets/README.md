# Asset directory

This directory holds the **current manifest asset** for each module, plus
its **asset-acceptance metadata**.

Today's contents:

| file | what it is | how it is made |
|---|---|---|
| `galley_1000.glb` | Current manifest asset for `galley_1000_sink_left_oak`. Today this is bit-identical to the golden contract fixture (a plain 1000×520×900 mm box anchored at the floor back-left corner with placeholder material slots and a placeholder collision proxy). Tomorrow it may be real cabinet art. See `galley_1000.glb.md`. | Generated deterministically by `tools/assets/generate_galley_fixture_glb.py` from `examples/galley_1000.json`. |
| `galley_1000.asset_acceptance.json` | Acceptance metadata for the manifest asset. Records the manifest it belongs to, whether the generated fixture has been replaced, the validator command, the required-checks gate list, and the human sign-off block. Validated by `tools/assets/validate_asset_acceptance.py`. | Hand-edited. The current file declares `asset_kind: "generated_contract_fixture"`, `generated_fixture_replaced: false`, and `human_signoff.production_art: false`. |

## Golden contract fixture vs. current manifest asset

Two GLBs participate in the contract, and they have different jobs:

- **Current manifest asset** — `examples/assets/galley_1000.glb`. This
  is the file every downstream consumer (UE5 importer, package report,
  runtime loader) reads. Today it is the generated box; tomorrow it
  may be real art. It must always **validate** against the manifest,
  but its **bytes are not pinned forever** — real art is allowed to
  replace it, gated by the acceptance metadata.

- **Golden contract fixture** — `tests/fixtures/galley_1000_contract_box.glb`.
  This is the **permanent regression reference**. Its bytes are pinned
  byte-for-byte to the generator's output. It does **not** appear as
  `visual.glb_path` in any manifest, and it stays exactly as it is
  forever, even after real cabinet art lands in the manifest asset
  slot. See `tests/fixtures/README.md` for the longer rationale.

Both files start out as the same generated bytes — that is the entire
point of having them co-exist today. The split lets us replace the
manifest asset later without weakening the regression test or losing
the known-good comparison target.

## Why these two files live in different directories

If the golden contract fixture lived next to the manifest asset under
`examples/assets/`, a future PR that "swaps in real art" would be
tempted (or accidentally allowed) to overwrite it. Keeping the golden
fixture under `tests/fixtures/` makes the rule mechanical: real art
replaces files under `examples/assets/`; nothing under `tests/fixtures/`
ever stops being the generated box.

## Adding a new module asset

1. Add the manifest entry under `examples/`.
2. Extend `tools/assets/generate_galley_fixture_glb.py` (or write a
   sibling script) so the new manifest produces a deterministic box.
   Each generator must stay stdlib-only and deterministic.
3. Commit both:
   - the manifest asset under `examples/assets/<id>.glb`, and
   - the golden contract fixture under
     `tests/fixtures/<id>_contract_box.glb`.
4. Add a sibling acceptance metadata file
   (`examples/assets/<id>.asset_acceptance.json`).
5. Add a regression test analogous to
   `test_golden_fixture_matches_generator_byte_for_byte`.

## Replacing the manifest asset with real cabinet art (future)

Not done in this PR. The procedure is recorded in
`tools/blender/EXPORT_REAL_ASSET.md`. The short version:

1. Build the GLB in Blender per `EXPORT_REAL_ASSET.md`. The export must
   pass `python tools/blender/check_asset_ready.py
   --manifest examples/galley_1000.json --glb /tmp/candidate.glb`.
2. Copy the candidate over `examples/assets/galley_1000.glb`. **Do
   not** touch `tests/fixtures/galley_1000_contract_box.glb` — that
   stays as the generated box.
3. Edit `examples/assets/galley_1000.asset_acceptance.json` and flip:
   - `generated_fixture_replaced: true`
   - `asset_kind`: from `"generated_contract_fixture"` to something
     like `"production_art"`
   - `human_signoff.visual_reviewed: true`
   - `human_signoff.production_art: true`
   - `human_signoff.reviewer`: name of the human who signed off
   - `human_signoff.notes`: what changed and when
4. Update `tools/assets/validate_asset_acceptance.py` (and its tests)
   so the new "real art" phase is the invariant instead of "generated
   contract fixture".
5. The byte-for-byte parity test
   (`test_manifest_asset_equals_golden_while_fixture_not_replaced`)
   becomes a no-op automatically — it short-circuits when the flag is
   flipped — but the golden fixture's own determinism test
   (`test_golden_fixture_matches_generator_byte_for_byte`) keeps
   running. That is the whole point of the split.

The day this happens, the assertions inside
`tools/assets/validate_asset_acceptance.py` for "false-for-now" flip
to "must be true" — the metadata's invariants progress with the
project's state.

No real art ships in PR #4. PR #4 is the fixture-swap mechanism only.
